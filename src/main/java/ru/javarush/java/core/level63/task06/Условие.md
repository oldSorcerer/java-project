
### Служба поддержки

Представьте, что вы работаете в службе поддержки огромного облачного сервиса. Каждый день к вам приходят тысячи запросов от пользователей. Когда возникает проблема, эти запросы "путешествуют" по множеству микросервисов, каждый из которых обрабатывается отдельным потоком. Как же отследить, что именно произошло с конкретным запросом от начала и до конца, если каждый сервис пишет свои логи?

Ваша миссия — внедрить "волшебную нить", которая свяжет все сообщения, относящиеся к одному запросу, позволяя легко проследить его путь через лабиринт вашей системы.

Реализуйте многопоточное приложение, где каждый поток имитирует обработку уникального запроса. Для этого каждый поток должен сгенерировать уникальный идентификатор (например, UUID) и добавить его в MDC (Mapped Diagnostic Context) с ключом "requestId".

Затем каждый поток должен записать в лог хотя бы одно сообщение, и в этом сообщении, благодаря MDC, должен автоматически присутствовать его уникальный "requestId". Это позволит вам легко отфильтровывать и анализировать логи, относящиеся к одному конкретному запросу, даже если он прошел через десятки потоков и сервисов.

#### Требования:
1. Приложение должно создавать и запускать несколько потоков, каждый из которых имитирует обработку отдельного запроса.
2. Каждый поток должен генерировать уникальный идентификатор запроса, например, с помощью UUID.
3. В каждом потоке уникальный идентификатор должен быть добавлен в MDC с ключом "requestId".
4. В каждом потоке должно быть записано хотя бы одно сообщение в лог, и в этом сообщении автоматически должен присутствовать "requestId" из MDC.
5. Конфигурация логгера должна обеспечивать, чтобы значение "requestId" из MDC автоматически подставлялось в каждую запись лога.
6. MDC для каждого потока должен быть независимым, чтобы идентификаторы запросов не смешивались между потоками.
7. После завершения обработки запроса поток должен очищать MDC от значения "requestId", чтобы избежать утечек контекста между задачами.
