
### Синхронизированная запись в общий лог-файл из нескольких потоков

Представьте, что вы разрабатываете центральную систему управления для оживлённой космической станции. Множество различных модулей — система жизнеобеспечения, навигация, энергетические установки — постоянно генерируют критически важные отчёты и предупреждения. Все эти сообщения должны попадать в единый, универсальный бортовой журнал station_events.log. Если несколько модулей попытаются записать информацию одновременно, в журнале воцарится хаос, и данные станут нечитаемыми.

Ваша задача — создать надёжный класс StationLogManager с единственным методом recordEvent(String eventMessage). Этот метод должен принимать любое сообщение о событии и аккуратно добавлять его в файл station_events.log, гарантируя, что каждое новое сообщение начинается с новой строки. Чтобы избежать путаницы и повреждения данных, крайне важно, чтобы метод recordEvent был синхронизирован таким образом, чтобы только один модуль мог записывать в журнал в каждый конкретный момент времени.

В главном центре управления (main методе) представьте, что вы запускаете пять независимых модулей станции, каждый из которых, охваченный энтузиазмом, пытается записать по десять своих уникальных статусов в бортовой журнал. Убедитесь, что после того, как все модули закончат свою работу, файл station_events.log будет содержать все пятьдесят сообщений, каждое на своей строке, в идеальном, читаемом порядке. Это залог безопасности и порядка на вашей космической станции!

#### Требования:
1. Метод recordEvent(String eventMessage) должен быть реализован таким образом, чтобы только один поток мог выполнять запись в файл station_events.log в любой момент времени.
2. Каждое сообщение, записываемое с помощью метода recordEvent, должно начинаться с новой строки в файле station_events.log.
3. В задаче должен быть реализован отдельный класс с именем StationLogManager, содержащий метод recordEvent(String eventMessage).
4. Метод recordEvent должен добавлять сообщения в файл station_events.log, не перезаписывая его содержимое (режим дозаписи).
5. В методе main необходимо создать и запустить пять независимых потоков (или Runnable/Thread), каждый из которых вызывает метод recordEvent десять раз с уникальными сообщениями.
6. После запуска всех потоков в main необходимо дождаться завершения их работы (например, с помощью join), чтобы гарантировать, что все сообщения будут записаны до проверки содержимого файла.
7. После завершения всех потоков файл station_events.log должен содержать ровно пятьдесят сообщений, каждое из которых располагается на отдельной строке.
8. В файле не должно быть перепутанных, частично записанных или повреждённых сообщений — каждое сообщение должно быть целым и читабельным.
