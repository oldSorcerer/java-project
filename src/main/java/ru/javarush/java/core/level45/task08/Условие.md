
### Обновление профилей пользователей: интеллектуальная миграция возраста

Представьте, что вы — разработчик системы управления профилями пользователей. Изначально, в первой версии вашей системы, информация о пользователе хранилась в классе UserV1. Этот класс имел поля userName типа String и userAgeInt типа int, и, конечно же, был Serializable с serialVersionUID равным 200L.

Для демонстрации, создайте объект UserV1 (например, "Alice", 30 лет) и сериализуйте его в файл с именем "userV1.dat". Это будет имитировать базу данных старой версии.

Со временем вы обнаружили, что формат хранения возраста как целого числа не всегда удобен; например, вы хотите отображать возраст как "30 лет", "неизвестно" или "младше 18". Поэтому во второй версии вашей системы вы решили обновить класс пользователя до UserV2. В этом новом классе поле userAgeInt было заменено на userAgeString типа String. Очень важно, что вы сохранили тот же самый serialVersionUID = 200L, чтобы система могла распознавать старые данные.

Ваша главная задача — обеспечить плавную миграцию данных. Вам нужно реализовать специальный метод readObject в классе UserV2. Этот метод должен уметь "читать" старое поле userAgeInt (которое было целым числом) из сериализованного потока и преобразовывать его в текстовое представление для нового поля userAgeString. Например, если старое userAgeInt было 30, новое userAgeString должно стать "30".

После реализации readObject, попробуйте десериализовать объект из файла "userV1.dat", но уже как UserV2. В конце, выведите значения полей десериализованного объекта в удобном формате, например: "Имя пользователя: <userName>, Возраст: <userAgeString>". Это покажет, как можно не только адаптировать класс к изменениям, но и активно переформатировать данные во время десериализации, обеспечивая бесшовный переход между версиями.

#### Требования:
1. Класс UserV1 должен реализовывать интерфейс Serializable и иметь serialVersionUID, равный 200L.
2. Программа должна создать объект UserV1 с конкретными значениями (например, "Alice", 30) и сериализовать его в файл "userV1.dat".
3. Класс UserV2 должен реализовывать интерфейс Serializable, иметь serialVersionUID = 200L и содержать поля userName типа String и userAgeString типа String (вместо userAgeInt).
4. В классе UserV2 должен быть реализован метод readObject, который при десериализации из потока корректно извлекает старое поле userAgeInt (если оно присутствует) и преобразует его в строку, присваивая полученное значение полю userAgeString.
5. Программа должна десериализовать объект из файла "userV1.dat" как экземпляр класса UserV2, используя реализованный механизм миграции данных.
6. После десериализации программа должна вывести значения полей userName и userAgeString в формате: "Имя пользователя: <userName>, Возраст: <userAgeString>".
